13/*
14 * WIDGETS - THEORY OF OPERATION AND NOTES
15 *
16 * Widgets are persistent view 'objects' for X-Plane. A widget is an object
17 * referenced by its opaque handle (widget ID) and the APIs in this file. You
18 * cannot access the widget's guts directly. Every Widget has the following
19 * intrinsic data:
20 *
21 * - A bounding box defined in global screen coordinates with 0,0 in the
22 * bottom left and +y = up, +x = right.
23 *
24 * - A visible box, which is the intersection of the bounding box with the
25 * widget's parents visible box.
26 *
27 * - Zero or one parent widgets. (Always zero if the widget is a root widget.
28 *
29 * - Zero or more child widgets.
30 *
31 * - Whether the widget is a root. Root widgets are the top level plugin
32 * windows.
33 *
34 * - Whether the widget is visible.
35 *
36 * - A text string descriptor, whose meaning varies from widget to widget.
37 *
38 * - An arbitrary set of 32 bit integral properties defined by 32-bit integral
39 * keys. This is how specific widgets
40 *
41 * store specific data.
42 *
43 * - A list of widget callbacks proc that implements the widgets behaviors.
44 *
45 * The Widgets library sends messages to widgets to request specific behaviors
46 * or notify the widget of things.
47 *
48 * Widgets may have more than one callback function, in which case messages
49 * are sent to the most recently added callback function until the message is
50 * handled. Messages may also be sent to parents or children; see the
51 * XPWidgetDefs.h header file for the different widget message dispatching
52 * functions. By adding a callback function to a window you can 'subclass' its
53 * behavior.
54 *
55 * A set of standard widgets are provided that serve common UI purposes. You
56 * can also customize or implement entirely custom widgets.
57 *
58 * Widgets are different than other view hierarchies (most notably Win32,
59 * which they bear a striking resemblance to) in the following ways:
60 *
61 * - Not all behavior can be patched. State that is managed by the XPWidgets
62 * DLL and not by individual widgets cannot be customized.
63 *
64 * - All coordinates are in global screen coordinates. Coordinates are not
65 * relative to an enclosing widget, nor are they relative to a display window.
66 *
67 *
68 * - Widget messages are always dispatched synchronously, and there is no
69 * concept of scheduling an update or a dirty region. Messages originate from
70 * X-Plane as the sim cycle goes by. Since x-plane is constantly redrawing, so
71 * are widgets; there is no need to mark a part of a widget as 'needing
72 * redrawing' because redrawing happens frequently whether the widget needs it
73 * or not.
74 *
75 * - Any widget may be a 'root' widget, causing it to be drawn; there is no
76 * relationship between widget class and rootness. Root widgets are imlemented
77 * as XPLMDisply windows.
78 *
79 */

95/*
96 * XPCreateWidget
97 *
98 * This function creates a new widget and returns the new widget's ID to you.
99 * If the widget creation fails for some reason, it returns NULL. Widget
100 * creation will fail either if you pass a bad class ID or if there is not
101 * adequate memory.
102 *
103 * Input Parameters:
104 *
105 * - Top, left, bottom, and right in global screen coordinates defining the
106 * widget's location on the screen.
107 *
108 * - inVisible is 1 if the widget should be drawn, 0 to start the widget as
109 * hidden.
110 *
111 * - inDescriptor is a null terminated string that will become the widget's
112 * descriptor.
113 *
114 * - inIsRoot is 1 if this is going to be a root widget, 0 if it will not be.
115 *
116 * - inContainer is the ID of this widget's container. It must be 0 for a root
117 * widget. for a non-root widget, pass the widget ID of the widget to place
118 * this widget within. If this widget is not going to start inside another
119 * widget, pass 0; this new widget will then just be floating off in space
120 * (and will not be drawn until it is placed in a widget.
121 *
122 * - inClass is the class of the widget to draw. Use one of the predefined
123 * class-IDs to create a standard widget.
124 *
125 * A note on widget embedding: a widget is only called (and will be drawn,
126 * etc.) if it is placed within a widget that will be called. Root widgets are
127 * always called. So it is possible to have whole chains of widgets that are
128 * simply not called. You can preconstruct widget trees and then place them
129 * into root widgets later to activate them if you wish.
130 *
131 */
132WIDGET_API XPWidgetID XPCreateWidget(
133 int inLeft,
134 int inTop,
135 int inRight,
136 int inBottom,
137 int inVisible,
138 const char * inDescriptor,
139 int inIsRoot,
140 XPWidgetID inContainer,
141 XPWidgetClass inClass);
