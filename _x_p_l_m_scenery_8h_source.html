25/***************************************************************************
26 * Terrain Y-Testing
27 ***************************************************************************/
28/*
29 * The Y-testing API allows you to locate the physical scenery mesh. This
30 * would be used to place dynamic graphics on top of the ground in a plausible
31 * way or do physics interactions.
32 *
33 * The Y-test API works via probe objects, which are allocated by your plugin
34 * and used to query terrain. Probe objects exist both to capture which
35 * algorithm you have requested (see probe types) and also to cache query
36 * information.
37 *
38 * Performance guidelines: It is generally faster to use the same probe for
39 * nearby points and different probes for different points. Try not to
40 * allocate more than "hundreds" of probes at most. Share probes if you need
41 * more. Generally, probing operations are expensive, and should be avoided
42 * via caching when possible.
43 *
44 * Y testing returns a location on the terrain, a normal vectory, and a
45 * velocity vector. The normal vector tells you the slope of the terrain at
46 * that point. The velocity vector tells you if that terrain is moving (and is
47 * in meters/second). For example, if your Y test hits the aircraft carrier
48 * deck, this tells you the velocity of that point on the deck.
49 *
50 * Note: the Y-testing API is limited to probing the loaded scenery area,
51 * which is approximately 300x300 km in X-Plane 9. Probes outside this area
52 * will return the height of a 0 MSL sphere.
53 *
54 */
55
56

176/***************************************************************************
177 * Magnetic Variation
178 ***************************************************************************/
179/*
180 * Use the magnetic variation (more properly, the "magnetic declination") API
181 * to find the offset of magnetic north from true north at a given latitude
182 * and longitude within the simulator.
183 *
184 * In the real world, the Earth's magnetic field is irregular, such that true
185 * north (the direction along a meridian toward the north pole) does not
186 * necessarily match what a magnetic compass shows as north.
187 *
188 * Using this API ensures that you present the same offsets to users as
189 * X-Plane's built-in instruments.
190 *
191 */
192

296/*
297 * XPLMLoadObject
298 *
299 * This routine loads an OBJ file and returns a handle to it. If X-Plane has
300 * already loaded the object, the handle to the existing object is returned.
301 * Do not assume you will get the same handle back twice, but do make sure to
302 * call unload once for every load to avoid "leaking" objects. The object will

304 *
305 * The path for the object must be relative to the X-System base folder. If
306 * the path is in the root of the X-System folder you may need to prepend ./
307 * to it; loading objects in the root of the X-System folder is STRONGLY
308 * discouraged - your plugin should not dump art resources in the root folder!
309 *
310 *
311 * XPLMLoadObject will return NULL if the object cannot be loaded (either
312 * because it is not found or the file is misformatted). This routine will
313 * load any object that can be used in the X-Plane scenery system.
314 *
315 * It is important that the datarefs an object uses for animation already be
316 * loaded before you load the object. For this reason it may be necessary to
317 * defer object loading until the sim has fully started.

349/*
350 * XPLMDrawObjects
351 *
352 * XPLMDrawObjects draws an object from an OBJ file one or more times. You
353 * pass in the object and an array of XPLMDrawInfo_t structs, one for each
354 * place you would like the object to be drawn.
355 *
356 * X-Plane will attempt to cull the objects based on LOD and visibility, and
357 * will pick the appropriate LOD.
358 *
359 * Lighting is a boolean; pass 1 to show the night version of object with
360 * night-only lights lit up. Pass 0 to show the daytime version of the object.
361 *
362 *
363 * earth_relative controls the coordinate system. If this is 1, the rotations
364 * you specify are applied to the object after its coordinate system is
365 * transformed from local to earth-relative coordinates -- that is, an object
366 * with no rotations will point toward true north and the Y axis will be up
367 * against gravity. If this is 0, the object is drawn with your rotations from
368 * local coordanates -- that is, an object with no rotations is drawn pointing
369 * down the -Z axis and the Y axis of the object matches the local coordinate
370 * Y axis.
371 *
372 */
373XPLM_API void XPLMDrawObjects(
374 XPLMObjectRef inObject,
375 int inCount,
376 XPLMDrawInfo_t * inLocations,
